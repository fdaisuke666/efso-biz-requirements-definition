01AS-IS — 現状の移動フローと責務

■ 移動の種別と現行の責務主体（3パターン）

パターン 1
倉庫間移動（WMS 連携）
Core/EFS BO が移動出荷指示を登録し、WMS へ定時送信。WMS が物理移動を実行し、実績を EFS へ報告。EFS が TRS（積送在庫）経由で論理在庫を更新。
✓ BO 画面から指示登録を確認済み
パターン 2
店間移動（在庫Ctrl 起点）
在庫Ctrl が移動指示を生成（DB-dbt-0127 カート登録）し、店舗 BO・外部調達へ配信。EFS/BO は移動実績の参照のみ。
パターン 3
店舗内ロケ変（GWCS 起点）
GWCS/POS からのロケ変操作（在庫異動 No.③）。EFS は LT3 内ロケ変更を受信して反映。EC 販売可否に影響なし。

■ 拠点間移動の 3 方式（EFS が定義する実装レベル）

方式概要制約
① 通常の移動出入荷 移動出荷 → TRS（積送）発生 → 移動先で入荷処理。出荷側・入荷側の両方で処理が必要 標準フロー。物流ロケ間の移動に対応
② 自動入荷設定ロケへの移動 移動出荷されると即座に移動入荷が自動作成。TRS を経由せず在庫が移動 移動先が非物流ロケに限定（物流ロケは NG）
③ 伝無入荷（伝票なし入荷） 出荷伝票なしで入荷処理のみを行い在庫を強制移動させる特殊方式 移動元が物流ロケの場合は NG
重要: EFS は移動時の数量・明細差異を認めていない設計制約があります（EFS-WMS 整合性保証のため）。差異発生時は WMS 側で調整実績を同時計上するか、移動出荷を取消し・やり直しで対応します（管理者権限）。この制約が DB 制約かアプリロジックかは未確認（TBD-31）。

04推奨方針
推奨 Option A — Core = 移動指示の発行 + 論理在庫の更新

根拠①（業務要件が確認済み）: AS-IS ドキュメントで「物流場所からの出荷（移動出荷を含む）は BO 画面から指示を行う」と明記されており、EFS/BO から移動出荷指示を登録する業務フローが現在も稼働中です。

根拠②（二重引当防止の仕組みが確立）: instructionsAllocatedStock（移動引当済数）により、移動指示中の在庫を受注引当から除外する仕組みが設計上確立されています。この仕組みを TO-BE でも活かせる Option A が合理的です。

根拠③（#9 との一貫性）: #9 で確定する「Core = 論理在庫 / WMS = 物理実行」の責務原則と完全に整合します。倉庫間移動は「Core が論理的な移動指示を持ち、WMS が物理的に実行する」という分業が自然です。

店間移動との整合: 店間移動は現行どおり在庫Ctrl（後継システム）が担当します。Core は倉庫間移動の指示に集中し、店間移動の実績を受信して論理在庫を反映する役割を維持します。
