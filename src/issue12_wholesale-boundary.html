02論点の整理
論点 A
EC 卸売の取引量
EC（EFS BO）経由の卸出荷/返品がどの程度存在するかが未調査。取引量がゼロまたは極小なら Core 側の卸売機能は不要で、GWCS に一本化できる。
論点 B
売上データの突合方式
Core（EC 売上 SoT）と GWCS（店舗売上・卸売管理）の売上データをどう突合・整合するか。月次の請求金額確定フローに影響する。
論点 C
#11 / #13 との連動
#11（仕入管理スコープ）の方針により卸売の仕入側処理が変わる。#13（店舗基幹売上 = GWCS 側で解決済）との整合性も必要。

░ 大方針（GWCS 存続確定による前提条件）

領域責務主体根拠
GWCS 卸売（店舗 POS 系）GWCSv0.6 で GWCS 存続確定。店舗系実績の集約は GWCS の責務
EC 卸売（BO 画面登録）CoreEC 起点の BtoB 取引は Core/注文ドメインまたはフルフィルメントの責務

05推奨方針
確定可Option A — Core に EC 卸売、GWCS に店舗系卸売を残す（役割分担を維持）

大方針: GWCS 存続確定・周辺改修なしの前提のもと、現行 EFS/GWCS 間の役割分担をそのまま引き継ぐ。EC 起点の BtoB 卸売は Core、店舗 POS 系経由の卸売集約は GWCS、卸売上 SoT は Coreという現行構造を変更しない。

根拠†（現行 EFS が主体で成立済み）: 卸売上は既に EFS が SoT。GWCS は店舗 POS 系実績の集約·通知役割に限定されており、両者は「重複」ではなく「役割分担」。新 Core は EFS 側の卸売機能をそのまま引き継げばよい（#11 と同一論理）。

根拠‡（#11 との整合）: #11 で仕入管理を Core に引き継ぐ方針と連動し、仕入→卸売上→月次請求を finance ドメイン内で完結。消化仕入の「売上+仕入同時計上」も Core 内で一貫処理できる。

根拠§（実装方針判断ブロッカーは消滅）: 取引量調査（B-1）は実装方針判断ではなく「Core 側卸売機能のサイジング判断」に降格。B-3（コード体系）、B-4（#11 連動）は設計フェーズで消化。方針 Fix を妨げる未解決ブロッカーは残っていない。

06確認事項リスト
  • #12CTO + 開発
    EC 経由の卸出荷/返品の取引量を調査。過去 12 か月の件数・金額を Biprogy に確認し、「ゼロ/極小」か「一定量存在」かを判定する
  • #12CTO + 開発
    在庫異動区分コード(071-082)の逆転可能性を efs-db/efs-bo の実装値で確認。卸(⑦)と外部EC(⑧)のコード割り当てを正確に特定する
  • #12CTO + 開発
    Core ↔ GWCS 間の売上データ突合方式を設計。月次の卸売上整合チェックの自動化方式を IL 設計に含める
  • #11CTO + 開発
    #11（仕入管理スコープ）の方針 Fix と #12 を同一セッションで判断。消化仕入の「売上+仕入同時計上」フローが卸売上にも影響するため整合確認
  • #12ビジネス
    外販/SaaS 展開時に BtoB 卸売機能が求められるかを確認。築地さん（WOS と外販の責任者）に方針を確認する
  • #12CTO + 開発
    卸売上の適格請求書（インボイス）発行要件を確認。Core で発行する場合のデータモデル設計を評価
07実装方針判断ブロッカーは全解消 — 残るのは設計フェーズ課題
前提の変化による解消: 「GWCS 存続確定」「周辺システム改修なし」の前提が固まり、#11 と同一論理で「EFS→Core 引継」方針が成立することが確認された。以前 Critical / High として挙げていたブロッカーは実装方針判断レベルでは解消し、以下 2 項目はサイジング／詳細設計フェーズに降格した。
