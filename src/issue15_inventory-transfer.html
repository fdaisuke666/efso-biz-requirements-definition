01AS-IS — 現状の移動フローと責務

■ 移動の種別と現行の責務主体（3パターン）

パターン 1
倉庫間移動（WMS 連携）
Core/EFS BO が移動出荷指示を登録し、WMS へ定時送信。WMS が物理移動を実行し、実績を EFS へ報告。EFS が TRS（積送在庫）経由で論理在庫を更新。
✓ BO 画面から指示登録を確認済み
パターン 2
店間移動（在庫Ctrl 起点）
在庫Ctrl が移動指示を生成（DB-dbt-0127 カート登録）し、店舗 BO・外部調達へ配信。EFS/BO は移動実績の参照のみ。
パターン 3
店舗内ロケ変（GWCS 起点）
GWCS/POS からのロケ変操作（在庫異動 No.③）。EFS は LT3 内ロケ変更を受信して反映。EC 販売可否に影響なし。

■ 拠点間移動の 3 方式

方式概要制約
① 通常の移動出入荷移動出荷 → TRS（積送）発生 → 移動先で入荷処理。標準フロー。物流ロケ間の移動に対応
② 自動入荷設定ロケへの移動移動出荷されると即座に移動入荷が自動作成。TRS を経由せず在庫が移動移動先が非物流ロケに限定（物流ロケは NG）
③ 伝無入荷（伝票なし入荷）出荷伝票なしで入荷処理のみを行い在庫を強制移動させる特殊方式移動元が物流ロケの場合は NG
重要: EFS は移動時の数量・明細差異を認めていない設計制約があります（EFS-WMS 整合性保証のため）。この制約が DB 制約かアプリロジックかは未確認（TBD-31）。

04推奨方針
推奨Option A — Core = 移動指示の発行 + 論理在庫の更新

根拠①（業務要件が確認済み）: AS-IS ドキュメントで「物流場所からの出荷（移動出荷を含む）は BO 画面から指示を行う」と明記。現在も稼働中。

根拠②（二重引当防止の仕組みが確立）: instructionsAllocatedStock（移動引当済数）により、移動指示中の在庫を受注引当から除外する仕組みが確立されています。

根拠③（#9 との一貫性）: #9 で確定する「Core = 論理在庫 / WMS = 物理実行」の責務原則と完全に整合します。

06決定を阻む不確定要素（ブロッカー）
前提: 本再整理は「周辺システム（WMS / 在庫Ctrl / FVL / GWCS / EFS BO 等）を本プロジェクトで改修しない」ことを前提にブロッカーを洗い直した結果です。

前提の固定により旧 B-1 / B-2 / B-3 は方針が自動確定（下記「前提により解消」参照）。残るブロッカーは継続 2 件・新たに顕在化するもの 2 件の計 4 件です。

■ 前提により解消（方針自動確定）

旧 B-1 → 解消
#9 との連鎖: WMS 未改修 → WMS が指示管理を担う Option B は物理的に不可 → Option A（Core = 指示発行 + 論理在庫）が一択で確定。#9 との同時判断は不要、確認ステップに降格。
旧 B-2 → 解消
BO 手動入力の継続可否: 他システム連携による自動化は周辺改修を伴うため不可 → BO からの手動入力を TO-BE でも継続が自動確定。BO 画面内で完結する自動化（Core 内機能追加）のみ余地あり。
旧 B-3 → 解消
在庫Ctrl の扱い: 在庫Ctrl を改修せず残す → 店間移動は在庫Ctrl 管轄で継続、Core は実績受信のみでスコープ確定。在庫Ctrl と Core の統合議論は本プロジェクトのスコープ外。

■ 残るブロッカー（優先度順）

B-6現行 IF（STELCD03 / STELCD19 等）のプロトコル互換を Core / IL のどちらで吸収するかHIGH
何が問題か: WMS を改修しない以上、現行の移動出荷指示 IF（STELCD03）・移動実績 IF（STELCD19）のプロトコル仕様（電文形式・項目・タイミング）は WMS 側の現行仕様のまま維持しなければならない。新 Core はこれをどこでどう受け止めるかの設計判断が未着手。

選択肢の例:
・IL（統合レイヤ）で現行電文を wrap し、Core は新 IF で通信する（Core は綺麗な新設計、IL が互換層）
・Core が直接現行 IF を保持する（IL を薄くするが Core に旧仕様が染み出す）
方針が未定のまま Core 設計に着手すると、IL の責務境界が決まらず Core と IL の担当範囲が曖昧になる。後から IL ↔ Core の IF を書き直すと大きな手戻りになる。
TO-BE の IL 設計方針（どこまで現行 IF を吸収するか）を決定し、STELCD03 / STELCD19 を含む移動系電文の扱いを先に確定する。#9 で議論する WMS ↔ Core の責務境界とセットで判断。
Owner: CTO + 開発 / 依存: TO-BE IL アーキテクチャ方針
B-7instructionsAllocatedStock の SoT を Core / 在庫Ctrl のどちらが持つかHIGH
何が問題か: 周辺改修なし前提で倉庫間移動 = Core、店間移動 = 在庫Ctrlと指示主体が分かれる。移動引当済数（instructionsAllocatedStock）で二重引当を防ぐ仕組みが両方にまたがるため、どちらが SoT（正本）として持つか、両者の同期方式が未定義。

具体的な懸念: 店間移動の指示が在庫Ctrl から出ているとき、Core 側の受注引当ロジックはその情報をどこから取るのか。リアルタイムに取れるのか、バッチでしか見えないのか。
同期方式を決めずに実装すると、受注引当と移動引当の整合が崩れて二重引当や引当漏れが発生する。現行 EFS でどう整合を取っているかの実装調査なしには再現できない。
現行 EFS で Core（倉庫間）と在庫Ctrl（店間）がどう引当情報を共有しているかを調査。TO-BE では Core を SoT として引き取るか、在庫Ctrl からの受信方式を維持するかを判断。
Owner: 開発チーム + 在庫Ctrl 担当 / 前提: 現行連携実装の実態調査が必要
B-4「移動数量のズレを認めない」ルールがシステムのどこで効いているか不明MEDIUM
ルールの内容: 倉庫 A から 100 個出荷したら、倉庫 B でも必ず 100 個入荷しなければならない制約が現行 EFS に存在する。

周辺改修なし前提での論点: WMS 側の挙動は変わらないため、Core は現行互換の制約を必ず維持する必要がある。柔軟化する選択肢はなく、「どこで実装するか」のみが論点。
・DB 制約（NOT NULL / CHECK 等）なら DDL を新 Core DB に移植
・アプリロジックなら該当処理を新 Core に移植
実装箇所を確認しないまま新 Core を設計すると、現行制約を再現できず EFS-WMS 整合性保証が崩れる。結果として移動実績の受け付けに失敗するリスク。
開発チームが現行 DDL を確認するだけで判明する。確認作業は半日〜1 日で完了。結果を open-decisions に記録し、Core 設計に反映する。
Owner: 開発チーム / 難易度: 低（即時確認可能）
B-5AUTO_ARRIVAL_PLACE 特例の使用実態（対象業務・ロケ・SKU）が未把握MEDIUM
特例の内容: 特定ロケへの移動で TRS（積送在庫）を経由せず即時入荷扱いにする特殊設定。本部や帳簿上の場所など、物理的な倉庫ではない場所向け。

周辺改修なし前提での論点: WMS / EFS BO の挙動は現行通り → 特例を残すことが事実上確定。論点は「新 Core のロケーション設計でこの特例をどう再現するか」と「対象の洗い出し」。
対象を洗い出さないまま新ロケーション設計を進めると、実運用で使われているロケが TRS を経由しない特例に該当することに後から気づくケースが発生し、リリース後の運用トラブルになる。
開発チーム + 倉庫運用チームが AUTO_ARRIVAL_PLACE を使用している業務・ロケ・SKU を洗い出し、新 Core のロケーション設計にマッピングする。
Owner: 開発チーム + 倉庫運用チーム

08補足 — 用語解説

本資料に登場する専門用語を「たとえ」付きで解説します。

移動出荷 / 移動入荷
倉庫Aから倉庫B（または店舗）へ商品を移すこと。出す側が「移動出荷」、受ける側が「移動入荷」。
たとえ: 本社倉庫から地方の支店に商品を送る社内便。顧客への販売ではなく社内の在庫移動。出荷と入荷で数が一致しなければならない。
TRS（積送在庫 / Transfer Stock）
移動出荷済みだがまだ移動入荷されていない「移動中」の在庫。出荷元からは減っているが、入荷先にはまだ届いていない状態。
たとえ: 銀行間の振込で、送金済みだが着金前の状態。お金は送り元の口座にも振込先の口座にも表示されない「宙に浮いた」状態。在庫管理ではこの宙ぶらりん在庫を正確に追跡することが重要。
instructionsAllocatedStock（指示済引当在庫）
移動指示が出されたが、まだ物理的に出荷されていない「予約済み」の在庫。二重に引き当てられることを防ぐ仕組み。
たとえ: レストランで「予約席」の札が置いてある席。まだお客さんは来ていないが、他のお客さんには案内できない。出荷指示 = 予約、実際の出荷 = 着席。
在庫Ctrl（在庫コントロール）
EC向けの在庫配分を管理する仕組み。どの販売チャネル（自社EC、モール等）にどれだけ在庫を割り当てるかを制御する。店間移動の指示も担当。
たとえ: パイを切り分ける係。倉庫にある 100 個の商品を「自社EC に 60、ZOZO に 30、楽天に 10」と配分する。TO-BE での役割が未決定（#15 のブロッカー B-3）。
WMS（倉庫管理システム）
倉庫内の入荷・保管・出荷を管理するシステム。棚の場所（ロケーション）単位で現物を追跡する。
たとえ: 巨大な図書館の蔵書管理システム。「どの棚の何段目に何冊あるか」を追跡する。Core とは別のシステムで、Core の指示に従って物理的な移動を実行する。
AUTO_ARRIVAL_PLACE（自動入荷設定ロケ）
特定のロケーション宛の移動で、TRS（積送在庫）を経由せず即時入荷扱いにする特殊設定。主に非物流ロケーション向け。
たとえ: 社内便で「この部署宛ては受取確認なしで即配達完了にする」という特例ルール。通常は受取確認が必要だが、特定の宛先だけ省略できる。
IL（統合レイヤー / Integration Layer）
TO-BE（新システム）で導入予定のシステム間通信を一元管理する基盤。現行の直接接続を整理する。
たとえ: 各部署が直接電話し合う状態から、交換台（IL）を置いてすべての連絡をそこ経由にする。Core → WMS の移動指示も IL 経由になる。
SoT（Source of Truth）
「数字が食い違ったとき、どのシステムの数字を正とするか」の取り決め。
たとえ: 戸籍が本籍地の役所にしかないのと同じ。コピーが複数あっても「正本」は 1 か所。移動在庫（TRS）の論理管理は Core が SoT。
