差異報告の標準化

05推奨方針
推奨Option A — 責務分離（Core = 論理在庫、WMS = 物理在庫）

前提条件: WMS は現行を据え置く方針のため、AS-IS の棚卸フロー（WMS 側で棚卸 → 差異を EFS に連携）は引き続き利用可能。TO-BE では IL 経由で標準化する。

根拠†（責務の明確化）: 「帳簿は Core、現場は WMS」と明確に分けることで、Core のスコープを抑制し保守性を確保します。現行 EFS でも論理在庫の SoT は EFS 側であり、この原則を踏襲します。

根拠‡（二重管理の解消）: GW テーブル（GW_LOCATION_STOCK）を廃止し、WMS との連携を IL 経由に一本化することで、現行の 2 テーブル構造を解消できます。

根拠§（差異報告の標準化）: 10 個の差異報告 API を IL の物流コネクタ（IL-5）で 1〜2 本に標準化します。商品識別キーの違いは IL 側のデータ変換で吸収します。

09補足 — 用語解説

本資料に登場する専門用語を「たとえ」付きで解説します。

EFS インスタンス
同じ EFS システムの別々のコピー（別環境）のこと。現行 EFS は 3 つの独立環境に分かれている。
たとえ: 同じ設計図で建てた 3 棟の別々のビル。見た目は同じだが、入居者（データ）は別々。
3 つのインスタンス: DS（データ配信元）/ EFS-WOS（自社 EC 用）/ EFS-FCL（モール出店用）
シングルインスタンス
3 つに分かれた EFS を1 つに統合すること。TO-BE（新システム）で検討中の方針。
たとえ: 3 棟のビルを解体して、1 棟の大きなビルに建て直すこと。統合すれば管理は楽になるが、引っ越し（データ移行）計画が必要。
論理在庫 / 物理在庫
論理在庫 = システム上の帳簿在庫（EFS が管理）。物理在庫 = 倉庫や店舗に実際にある現物の数。
たとえ: 銀行口座の残高表示（論理在庫）と、実際に金庫にある現金（物理在庫）。通常は一致するが、ズレることがある。
SoT（Source of Truth）
「在庫の数が食い違ったとき、どのシステムの数字を正とするか」の取り決め。
たとえ: 戸籍が本籍地の役所にしかないのと同じ。コピーが複数あっても「正本」は 1 か所。現行 EFS では論理在庫の SoT は EFS 側が持つ。
WMS（倉庫管理システム）
倉庫内の入荷・保管・出荷を管理するシステム。棚の場所（ロケーション）単位で現物を追跡する。
たとえ: 巨大な図書館の蔵書管理システム。「どの棚の何段目に何冊あるか」を追跡する。EFS とは別システム。
GW（WMS Gateway）
EFS と WMS の間を繋ぐ中継システム。EFS 内部に GW 専用のテーブル（データの入れ物）が存在する。
たとえ: 2 つのビル（EFS と WMS）の間にある連絡通路。荷物（データ）はこの通路を通ってやり取りされる。
IL（統合レイヤー / Integration Layer）
TO-BE（新システム）で導入予定のシステム間通信を一元管理する基盤。現行のスパゲッティ状の接続を整理する。
たとえ: 各部署が直接電話し合う状態から、交換台（IL）を置いてすべての連絡をそこ経由にする。監視・変換・再送が一元管理できる。
差異報告 API
WMS の物理在庫と EFS の論理在庫にズレがないか検知するための通信口。現行は 10 個も乱立している。
たとえ: 銀行と家計簿の残高が合っているか確認する「残高照合」を、10 種類の異なるフォーマットでやっている状態。TO-BE では IL で標準化する。
