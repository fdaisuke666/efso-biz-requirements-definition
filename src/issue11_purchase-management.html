02論点の整理
論点 A
機能の重複
EFS と GWCS の仕入関連機能が一部重複。GWCS 廃止時にどちらに集約するかを先に決めないと仕入機能が宙に浮く。
論点 B
EDI 連携（IL-19）
Option A（Core 化）の場合、取引先 EDI（IL-19）が必要。Core が発注ワークフローを持つなら IL 経由で EDI 接続を設計する必要がある。
論点 C
#23 ドメイン分離
#23（売上/仕入の独立ドメイン化）と連動。finance を独立ドメインにするかどうかの方針と整合が必要。

05推奨方針
推奨Option A — EFSO Core に仕入管理を含める

根拠①（GWCS 廃止との整合）: GWCS は段階的廃止フェーズにあります。仕入管理を GWCS に残すと廃止のブロッカーになり、中長期で技術負債が増大します。

根拠②（現行設計の自然な延長）: EFS は既に発注管理・月次締め・支払管理の主要機能を保持しており、Core 化は現行設計の自然な延長です。

根拠③（#23 との連動）: finance を独立ドメインとして分離する方向で #23 の方針 Fix と同時に判断することで、整合した設計が可能になります。

07決定を阻む不確定要素（ブロッカー）

以下の5項目が未解決のまま残ると、Option A を採用しても詳細設計・移行計画の策定に着手できません。優先度順に解消してください。

B-1GWCS 側に残っている仕入機能の全体像が把握できていないCRITICAL
何が問題か: 現行は仕入関連の機能が EFS と GWCS の両方に分散している。既に判明している「外注入荷報告（仕入）」以外にも、GWCS 側で動いている仕入機能があるかもしれないが、誰もその全体像を把握していない。

具体例: 発注書の作成画面、支払予定の登録画面、仕入先マスタの保守画面などが GWCS にあるかもしれない。
「どこからどこまで Core に移すのか」が決まらないため、工数見積もりができず、プロジェクトのスケジュールが組めない。調査漏れの機能があった場合、移行後に「この機能が動かない」という本番障害に直結する。
2 週間以内に GWCS の仕入関連機能を棚卸し、「EFS にある機能」「GWCS にある機能」「両方に重複している機能」を一覧化する。
Owner: CTO + 開発チーム / 期間: 2 週間以内
B-2取引先との電子データ連携（EDI）が本当に作れるか不明CRITICAL
EDI とは: 発注書・納品書・請求書などを紙や FAX ではなく、システム間で直接データをやり取りする仕組み。取引先（仕入先メーカー等）と自社システムをつなぐ専用の通信経路が必要。

何が問題か: Option A（Core に仕入管理を含める）を選ぶと、取引先 EDI（IL-19 と呼ばれる統合レイヤーの1つ）を新たに作る必要がある。しかし「接続先の取引先がどんなフォーマット・プロトコルを要求するか」「取引先側がこちらの仕様変更に付き合ってくれるか」が未確認。
EDI 接続先が多い・技術的に難しい・取引先の協力が得られない場合、開発コストが想定の数倍になる可能性がある。場合によっては Option A 自体を諦めて Option B（GWCS に残す）に戻す判断が必要になる。
主要な取引先 2〜3 社と話し、(1) 現在使っている EDI のフォーマット (2) Core 移行に伴う仕様変更に対応可能か (3) 接続方式（専用線/インターネット/クラウドサービス経由）を確認する。
Owner: CTO + 開発チーム / 連携: 調達
B-3「売上仕入」を独立ドメインにするかの方針が未決定（#23 と連動）HIGH
何が問題か: 別課題 #23 で「売上/仕入（finance）を 1 つの独立した大きなドメインとして設計するか、受注ドメインの中に含めたままにするか」が未決定。仕入管理（#11）は売上仕入の一部なので、#23 の方針次第で Core 内の置き場所が変わる。

具体例:
・#23 で「独立ドメイン」確定 → 仕入管理は finance ドメインの下に配置
・#23 で「受注ドメイン統合」確定 → 仕入管理は受注ドメインの下に配置
#11 だけ先に決めて設計を始めると、#23 の結論と矛盾して、あとから大幅な作り直しが発生する。テーブル設計・API 設計・画面設計すべてに影響が出る。
#11 と #23 を同じアーキテクチャ判断会で同時に決定する。CTO がオーナーとなり、開発チームと判断会を設定する。
Owner: CTO + 開発
B-4消化仕入（売れたら仕入計上）の仕組みが Core で再現できるか未検証HIGH
消化仕入とは: メーカーから商品を受け取った時点では仕入にせず、顧客に売れた瞬間に初めて「売上」と「仕入」を同時に計上する取引形態。売れ残りは自社の在庫にならずメーカーに返す。アパレルでは一般的。

何が問題か: 通常の買取仕入（先に発注→入荷→在庫→売上）とはフローが全く異なり、「発注なし・出荷イベントで売上と仕入を同時計上」という特殊処理が必要。この特殊ロジックが現行 EFS にあるが、新 Core で同じように動かせるかが設計レベルで未確認。
アパレルでは消化仕入の比率が高いため、このフローが動かないと売上・仕入データが狂い、月次決算の数字が合わなくなる。発見が遅れると本番障害になる。
開発チームが消化仕入の現行ロジックを読み込み、EFSO Core のデータモデルに移植できるかを設計レビューする。移植困難な部分があれば TO-BE 設計書に影響を反映する。
Owner: CTO + 開発チーム
B-5アットトバス（支払報告書の郵送代行サービス）を今後も使うか未決定MEDIUM
アットトバスとは: 仕入先への支払報告書（「今月はこれだけお支払いします」の明細書）を印刷して郵送する代行サービスを提供する外部ベンダー。EFS は現在このサービスにデータを渡して郵送を任せている。

何が問題か: 新システムでもこの郵送代行を使い続けるのか、電子メール配信や取引先向けポータルに切り替えるのかがビジネス側で決まっていない。外部ベンダーとの契約・連携仕様が Core 側の設計スコープに直接影響する。
継続する場合は統合レイヤー（IL-19）にアットトバス連携を含める必要があり、設計規模が増える。廃止する場合は電子配信の新規設計が必要。どちらにせよ「決まらないと設計に着手できない」領域。
調達・ビジネスチームで (1) 郵送継続が業務上必須か (2) 電子化の合意が取引先から得られるか (3) 契約更新時期はいつか を確認し、TO-BE 方針を決定する。
Owner: ビジネス・調達 / 連携: CTO

09補足 — 用語解説

本資料に登場する専門用語を「たとえ」付きで解説します。

仕入管理
商品を仕入先から買う（または預かる）一連の業務。発注・入荷・検収・支払・請求までを含む。
たとえ: レストランが食材を業者から仕入れるのと同じ流れ。「何を何個発注→届いたか確認→今月いくら払うか計算→支払」の一連の業務。EC では売上を支える根幹業務。
買取仕入 / 消化仕入
買取仕入 = 先に買い取って在庫にする通常の仕入。
消化仕入 = 商品を預かり、売れた分だけ仕入計上する仕入形態。売れ残りはメーカーに返品。
たとえ:
・買取仕入 = スーパーで野菜を 100 個買って店頭に並べる（売れなくても自腹）
・消化仕入 = 委託販売の雑貨屋。お店にあるが売れた分だけ作家さんにお金を払う。アパレルでは消化仕入の比率が高い。
EDI（電子データ交換）
取引先との間で発注・納品・請求などのデータを電子的にやり取りする仕組み。紙・FAX・メール添付ではなく、システム間で直接連携する。
たとえ: FAX で発注書を送っていたのを、システム同士で自動的にデータを送受信する仕組みに変えたもの。人手が要らず、速く正確。取引先ごとにフォーマットが違うのが難点。
IL-19（取引先 EDI コネクタ）
新システムで想定されている取引先 EDI 用の統合レイヤー（IL）機能の識別番号。取引先ごとのフォーマット差異を吸収する役割。
たとえ: 各取引先が別々の言語を話す状態を、通訳（IL-19）を間に置いて Core が 1 つの共通言語で話せばよいようにする仕組み。
GWCS（店舗/倉庫基幹システム）
店舗・倉庫の業務を担う現行の基幹系システム。POS 売上集約、店舗棚卸、配分指示、在庫予算、管理会計など広範な機能を持つ。存続が確定している。
たとえ: 店舗・倉庫側の「司令塔」。EC 側の EFS（→ Core）と役割分担していて、今回は仕入機能をどちらに寄せるかが論点。
外注入荷報告（仕入）
外部倉庫や仕入先メーカーから「入荷が完了しました」という報告を受け取り、仕入として計上する処理。現行は GWCS 側で実施。
たとえ: 外部倉庫の担当者が「荷物を受け取りました」という報告書を本社に送るイメージ。この報告をトリガーにして仕入計上が走る。
アットトバス（支払報告書郵送サービス）
仕入先への「今月はこれだけお支払いします」という支払報告書を印刷・郵送代行する外部サービス。EFS がデータを連携して郵送を委託している。
たとえ: 自社で印刷・封入・切手貼り・投函をやる代わりに、外部業者に一括委託している状態。電子メール送付やポータル掲載に切り替えるかが論点（B-5）。
finance ドメイン / 売上仕入
売上計上・仕入計上・請求・支払などお金の流れを扱う業務領域のまとまり。#23 で独立ドメインにするかが議論中。
たとえ: 会社の「経理部門」に相当する機能群。現在は受注ドメインの中に混在しているが、独立した部門（ドメイン）として切り出すかを検討中。#11 の配置先が変わる。
