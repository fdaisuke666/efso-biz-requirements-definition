重要: 卸出荷/返品は POS 系経由（GWCS 集約）のルートと、EFS BO 画面からの直接登録ルートの 2 経路が存在する。EC 経由の卸売取引量が不明なため、Core 側にどの程度の卸売機能を持たせるべきかの判断が保留されている。
02論点の整理
論点 A
EC 卸売の取引量
EC（EFS BO）経由の卸出荷/返品がどの程度存在するかが未調査。取引量がゼロまたは極小なら Core 側の卸売機能は不要で、GWCS に一本化できる。
論点 B
売上データの突合方式
Core（EC 売上 SoT）と GWCS（店舗売上・卸売管理）の売上データをどう突合・整合するか。月次の請求金額確定フローに影響する。
論点 C
#11 / #13 との連動
#11（仕入管理スコープ）の方針により卸売の仕入側処理が変わる。#13（店舗基幹売上 = GWCS 側で解決済）との整合性も必要。

開発コスト▵ 卸出荷/返品 + 請求管理の実装が必要◎ 卸売機能の新規開発が不要
05推奨方針
推奨Option A — Core に EC 卸売を残す（条件付き）

前提条件: EC 経由の卸売取引量が「ゼロではない」ことが確認された場合。取引量調査の結果、完全にゼロであれば Option B を採用する。

根拠†（SoT 一元性）: 現行 EFS が卸売上の SoT であり、この一元性を崩すと月次の売上突合フローが複雑化します。Core に卸売機能を残すことで、EC 売上（BtoC）と EC 卸売上（BtoB）を同一ドメインで管理できます。

根拠‡（#11 との整合）: #11 で仕入管理を Core に含める方針と連動し、仕入→卸売上→月次請求の一連のフローを finance ドメイン内で完結させることで、アーキテクチャの整合性が保たれます。

根拠§（外販展開）: next-EFS は外販を前提とした EC システム。BtoB 卸売機能は外販先テナントにも求められる可能性が高く、Core に持たせる合理性があります。

07決定を阻む不確定要素（ブロッカー）

以下の 4 項目が未解決のまま残ると、Option A/B いずれを選択するかの最終判断ができません。

B-1EC 卸売の取引量が未調査CRITICAL
EC（EFS BO）経由の卸出荷/返品件数が不明。この数値が Option A/B の選択を直接決定する最重要ファクター。
取引量なしに方針を決めると、過剰実装（量ゼロなのに卸売機能を作る）または機能欠落（量があるのに GWCS 一本化）のリスク。
Biprogy に過去 12 か月の EC 卸出荷/返品の件数・金額データを依頼。2 週間以内にデータを取得する。
Owner: CTO + 開発 / 確認先: Biprogy（此松さん、長嶋 春奈さん）
B-2売上データ突合方式が未設計HIGH
Core（EC 売上 SoT）と GWCS（店舗/卸売実績）の売上データ突合方式が未設計。月次締め時の整合チェックをどのレイヤーで実行するかが不明。
突合方式が決まらないと、Core ↔ GWCS 間の IL 設計が確定できず、FS-6（GWCS 売上連携）の要件定義が遅延する。
売上突合の要件（頻度・粒度・不一致時のエスカレーション）を定義し、IL 設計に反映。
Owner: CTO + 開発
B-3在庫異動区分コードの逆転可能性HIGH
ドキュメント上は区分⑦=卸、⑧=外部EC だが、CodeConsts では 071-072=外部EC、081-082=卸 と逆転の可能性がある。データ移行時に売上区分を誤ると財務データが破損する。
コード体系の取り違えにより、卸売上と外部 EC 売上のデータが混在し、月次決算の正確性に影響。
efs-db/efs-bo の実装コードを確認し、正しいコード体系を特定。ドキュメントとの乖離があれば修正する。
Owner: 開発チーム / 確認先: Biprogy
B-4#11 仕入管理スコープとの連動判断MEDIUM
#11 で仕入管理を Core に含めるか GWCS に残すかの方針が卸売の処理フローに影響。消化仕入商品の場合、卸出荷時に「売上+仕入の同時計上」が発生するため、仕入側の SoT と卸売上側の SoT を同一システムに置く必要がある。
#11 と #12 を別々に判断すると、消化仕入フローが Core と GWCS に分断され、同時計上ロジックの整合性が崩れるリスク。
#11 と #12 を同一の意思決定セッションで方針 Fix する。CTO がオーナーとなり判断会を設定。
Owner: CTO + 開発 / 連携: 長谷川さん、築地さん
08TO-BE 推奨アプローチ & ネクストアクション
推奨方針は「取引量調査の結果に基づき Option A（Core に EC 卸売を残す）を基本路線とする」。
ただし B-1（取引量調査）の結果が「完全ゼロ」であれば Option B に切り替える。まず取引量データの取得が最優先タスク。
