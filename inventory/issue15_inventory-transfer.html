01AS-IS — 現状の移動フローと責務

■ 移動の種別と現行の責務主体（3パターン）

パターン 1
倉庫間移動（WMS 連携）
Core/EFS BO が移動出荷指示を登録し、WMS へ定時送信。WMS が物理移動を実行し、実績を EFS へ報告。EFS が TRS（積送在庫）経由で論理在庫を更新。
✓ BO 画面から指示登録を確認済み
パターン 2
店間移動（在庫Ctrl 起点）
在庫Ctrl が移動指示を生成（DB-dbt-0127 カート登録）し、店舗 BO・外部調達へ配信。EFS/BO は移動実績の参照のみ。
パターン 3
店舗内ロケ変（GWCS 起点）
GWCS/POS からのロケ変操作（在庫異動 No.③）。EFS は LT3 内ロケ変更を受信して反映。EC 販売可否に影響なし。

■ 拠点間移動の 3 方式

方式概要制約
① 通常の移動出入荷移動出荷 → TRS（積送）発生 → 移動先で入荷処理。標準フロー。物流ロケ間の移動に対応
② 自動入荷設定ロケへの移動移動出荷されると即座に移動入荷が自動作成。TRS を経由せず在庫が移動移動先が非物流ロケに限定（物流ロケは NG）
③ 伝無入荷（伝票なし入荷）出荷伝票なしで入荷処理のみを行い在庫を強制移動させる特殊方式移動元が物流ロケの場合は NG
重要: EFS は移動時の数量・明細差異を認めていない設計制約があります（EFS-WMS 整合性保証のため）。この制約が DB 制約かアプリロジックかは未確認（TBD-31）。

04推奨方針
推奨Option A — Core = 移動指示の発行 + 論理在庫の更新

根拠①（業務要件が確認済み）: AS-IS ドキュメントで「物流場所からの出荷（移動出荷を含む）は BO 画面から指示を行う」と明記。現在も稼働中。

根拠②（二重引当防止の仕組みが確立）: instructionsAllocatedStock（移動引当済数）により、移動指示中の在庫を受注引当から除外する仕組みが確立されています。

根拠③（#9 との一貫性）: #9 で確定する「Core = 論理在庫 / WMS = 物理実行」の責務原則と完全に整合します。

06決定を阻む不確定要素（ブロッカー）

以下の5項目が未解決のまま残ると、Option A を採用しても TO-BE の詳細設計に着手できません。特に B-1（#9 との連鎖）が最優先です。

B-1#9 との責務境界が連鎖している（同時判断が必要）CRITICAL
#15 の「Core が移動指示を発行する」という結論は、#9 の「Core = 論理在庫 / WMS = 物理実行」という責務原則に基づいている。#9 が確定しないと #15 の根拠が固まらない。
#9 と #15 を別々のタイミングで判断すると、整合性が崩れた設計になるリスクがある。
#9 と #15 を必ず同一のアーキテクチャ判断セッションで方針 Fix する。CTO + 開発がオーナーとなり日程を設定。
Owner: CTO + 開発 / 対応: #9 の B-1 解消後に同一セッションで判断
