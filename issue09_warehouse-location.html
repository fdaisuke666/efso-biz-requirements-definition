論理在庫と物理在庫の責務を Core と WMS のどちらに置くか

01AS-IS — 現状のシステム構成

■ 在庫管理の概念区分

論理在庫（Logical Inventory）
EFS/Core が SoT。 管理単位:「場所ID × 場所属性ID」。棚番・ケース単位の管理は行わない（原本明記）。L1/L2/L3 引当階層・TRS（積送在庫）を含む。
物理在庫（Physical Inventory）
WMS が SoT。 管理単位: 棚番（実ロケーション）レベル。入荷検品・出荷検品・棚卸・差異報告の起点。B在庫 / C在庫を保持。

05推奨方針
推奨Option A — Core = 論理在庫、WMS = 物理在庫 + 棚卸

根拠①: AS-IS ドキュメントで「EFS は棚の管理を行わない・ケースの管理を行わない」と明記。現行設計が既に Option A の構造で稼働中。

根拠②: WMS は棚卸ケース1〜3をバッチ含め実装・稼働済み。廃棄処理確定を EFS が担う合理的な分業が成立。

根拠③: Option B は現行 DDL に棚番管理カラムが存在しないため、EFSO プロジェクトのスコープ拡大リスクが高い。

07決定を阻む不確定要素（ブロッカー）

以下の4項目が未解決のまま残ると、Option A を採用しても TO-BE の詳細設計に着手できません。各ブロッカーを解消するための具体的アクションを次のセクションに記載します。

B-1 EFSインスタンス統合方針が未決定 CRITICAL
現行の EFS は DS / WOS / FCL の3インスタンスに分散しており、それぞれが論理在庫の SoT 権限を持つ可能性がある。EFSO Core がシングルインスタンスになるかどうかが決まっていない（OI-C01 未解決）。
TO-BE のデータモデル設計が確定できない。インスタンスごとにデータ構造が変わる場合、Core 化のスコープが大幅に広がる。
OI-C01 を EFSO プロジェクトの前提条件として先行 Fix する。CTO + 開発が判断。
Owner: CTO + 開発チーム
B-2 在庫テーブルの二重管理（GW_LOCATION_STOCK / LOCATION_STOCK） HIGH
場所区分05（GW管理）は LOCATION_STOCK とは別の GW_LOCATION_STOCK テーブルで管理されており、どちらを正とするかが TO-BE で決まっていない。
Core 側の在庫管理テーブル設計が分岐し、整合性確保のコストが増大する。
TO-BE では LOCATION_STOCK に一本化する方針で合意を取る。GW_LOCATION_STOCK は移行フェーズで廃止。
Owner: CTO + 開発チーム
B-3 差異報告APIの乱立（GW系 + v1/v2/v3系 計10エンドポイント） HIGH
WMS→Core 間の差異報告 API が GW 系と v1/v2/v3 系に分散して並存している。TO-BE の IL（統合レイヤー）でどのAPIを標準とするかが決まっていない（OI-E02）。
IL 設計時に全パターンの調査が必要になり、移行コストが予測できない。
IL 移行を機に v1 系 API に標準化する方針で開発チーム内合意を取る。旧 GW 系 API は廃止スケジュールを確定。
Owner: 開発チーム
B-4 各倉庫の棚卸標準フローが未確認 MEDIUM
茜浜・流山・舞浜の各倉庫で実際にケース1/2/3のどのフローが使われているかが現地確認されていない。倉庫ごとにフローが異なる可能性がある。
TO-BE で棚卸フローを統一すべきかどうかの判断ができず、WMS 連携仕様が確定しない。
各倉庫担当者へのヒアリングを実施（目安1週間）。フロー差異がある場合は TO-BE で統一するか個別対応するかを判断。
Owner: 倉庫運用チーム
